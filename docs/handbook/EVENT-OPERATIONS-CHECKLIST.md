# LEAD Talent Platform - Event Operations Checklist

Esta guia es para chapter leaders y chapter editors que van a operar eventos en LEAD Talent Platform. La meta es que cada evento tenga una configuracion clara, evidencia de asistencia y una base util para futuros Impact Metrics.

## Antes de crear el evento

- Confirmar el objetivo del evento: que problema resuelve, para quien es y que resultado esperamos.
- Confirmar owner del evento: chapter responsable, chapter editor y persona que hara seguimiento.
- Definir si el evento sera abierto o con aplicacion.
- Confirmar capacidad, fecha, hora, ubicacion o link virtual.
- Confirmar si habra partners, speakers o empresas invitadas.

## Configuracion del evento

- Usar titulo claro y consistente con la comunicacion oficial.
- Agregar descripcion breve con proposito, audiencia y expectativa.
- Confirmar `event_type`: presencial, online o hibrido.
- Confirmar `access_model`: open o application.
- Publicar solo cuando el contenido principal ya este revisado.
- Revisar que el evento aparezca en eventos publicos despues de publicarlo.

## Evento abierto

Usar evento abierto cuando cualquier participante puede registrarse directamente.

- Validar que una persona puede registrarse aunque no pertenezca a un chapter.
- Confirmar que el registro genera evidencia de asistencia esperada.
- Usar este modelo para eventos informativos, workshops generales o actividades con cupos amplios.

## Evento con aplicacion

Usar evento con aplicacion cuando el chapter necesita revisar respuestas antes de aprobar asistencia.

- Mantener preguntas cortas y accionables.
- Marcar como requeridas solo las preguntas realmente necesarias.
- Evitar pedir informacion sensible si no es necesaria.
- Revisar que las respuestas se vean correctamente antes de abrir el evento.
- Definir quien aprueba o rechaza aplicaciones y en que fecha.

## Revision de aplicaciones

- Revisar aplicaciones desde el chapter dashboard.
- Aprobar solo cuando la persona debe quedar registrada para asistir.
- Rechazar cuando la persona no cumple el criterio o el cupo ya no permite mas aprobaciones.
- Verificar que las aplicaciones aprobadas pasen a registro confirmado.
- Documentar cualquier criterio especial fuera de la plataforma si todavia no existe como campo.

## Check-in el dia del evento

- Definir quien estara a cargo del check-in antes del evento.
- Probar el flujo de check-in antes de recibir asistentes.
- Usar QR o busqueda de asistentes segun el caso.
- Confirmar que una persona registrada pasa a estado attended despues del check-in.
- No hacer check-in manual sin razon operativa clara.

## Evidencia post-evento

- Revisar registrados, asistentes reales y tasa de asistencia.
- Guardar observaciones importantes: que funciono, que mejoraria y si el evento deberia repetirse.
- Identificar si hubo partner, speaker, sponsor o empresa relacionada.
- Usar la evidencia de check-in como base para futuros Impact Metrics.

## Escalacion y soporte

- Si el evento no aparece publicado, revisar `is_published`, fecha y chapter owner.
- Si una persona no puede registrarse, revisar si ya tiene registro activo o si el evento ya inicio.
- Si una aplicacion no se puede aprobar, revisar capacidad y estado de la aplicacion.
- Si check-in falla, confirmar que la persona esta registrada y no solo en pending_review.
- Escalar problemas tecnicos con captura, usuario de prueba/rol, evento afectado y pasos para reproducir.

## Recordatorio operativo

La plataforma permite que participantes publicos se registren o apliquen a eventos sin ser miembros aprobados de un chapter. Eso es intencional: nos ayuda a abrir la puerta a nuevos participantes y medir interes real antes de una membresia formal.
